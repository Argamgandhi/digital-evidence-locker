=======================================================================

The project is a tamper-proof digital evidence management system where:
- Files are hashed using SHA-256
- Stored on IPFS via Pinata
- Hash is permanently recorded on Ethereum Sepolia testnet via a Solidity smart contract
- Anyone can verify a file's authenticity by re-computing its hash and checking blockchain

  sequelize, sqlite3, axios, form-data
- Blockchain: Solidity ^0.8.20 on Ethereum Sepolia via Hardhat v2
- Storage: IPFS via Pinata API
- Auth: JWT with roles (personal/professional/organization/verification_authority)

- sets status to Submitted
- emits EvidenceSubmitted(fileHash, uploadedBy, timestamp)

- only AUTHORITY_ROLE
- authority cannot vote twice on same evidence (check mapping)
- if approve: add to approvedBy array
- if reject: add to rejectedBy array, store reason
- after each vote, check consensus:

  * if rejectedBy.length >= requiredRejections → set status = Rejected,

- emits VoteCast(fileHash, authority, approve, timestamp)

- EvidenceVerified(string fileHash, uint256 approvalCount, uint256 timestamp)
- EvidenceRejected(string fileHash, uint256 rejectionCount, string reason, uint256 timestamp)
- ConsensusThresholdUpdated(uint256 newApprovals, uint256 newRejections)

FEATURE 5: GAS OPTIMIZATION IN SMART CONTRACT
-------------------------------------------------------------------

Apply these gas optimization techniques to both EvidenceLocker.sol 
and EvidenceToken.sol:

1. Use bytes32 instead of string for fileHash in mappings:
   - mapping(bytes32 => Evidence) private evidenceStore
   - All internal logic uses bytes32, but external functions accept 
     string and convert using keccak256(bytes(fileHash))
   - Keep string in the Evidence struct for readability in getters

2. Use uint128 for timestamp and vote counts inside struct 
   instead of uint256 to pack storage slots:
   struct Evidence {
     string fileName;
     string fileHash;
     string ipfsCID;
     address uploadedBy;      // 20 bytes
     uint128 timestamp;       // 16 bytes — packed with uploadedBy
     uint128 lastUpdated;     // 16 bytes
     uint16 approvalCount;    // 2 bytes
     uint16 rejectionCount;   // 2 bytes
     EvidenceStatus status;   // 1 byte enum
     bool exists;             // 1 byte — packed together
     string description;
     string rejectionReason;
     address[] approvedBy;
     address[] rejectedBy;
   }

3. Use custom errors instead of require strings:
   error EvidenceAlreadyExists(string fileHash);
   error EvidenceNotFound(string fileHash);
   error AlreadyVoted(address authority, string fileHash);
   error NotAuthorized(address caller);
   error InvalidConsensusThreshold();
   Replace all require("string message") with revert CustomError(...)

4. Add view modifier to all read-only functions

5. Use unchecked {} blocks for safe increments inside vote counting 
   loops to save gas on overflow checks

6. Emit events before state changes where possible to save gas 
   on reverting transactions

7. In getAllHashes() and getAllPendingEvidence(), avoid returning 
   the full array in one call — instead add pagination:
   function getHashesPaginated(uint256 start, uint256 count) 
     external view returns (string[] memory)

8. Add a natspec comment above each function explaining gas cost 
   considerations like:
   /// @dev Uses bytes32 mapping key to save ~500 gas vs string key lookup

Add a file: blockchain/GAS_OPTIMIZATION_NOTES.md documenting:
- Before/after estimated gas costs for each major function
- Which optimizations were applied and why
- How to measure gas using Hardhat gas reporter
- Add hardhat-gas-reporter to hardhat.config.js
- Table format showing: Function | Unoptimized Gas | Optimized Gas | Saving %
